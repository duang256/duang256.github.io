0.Java结构

write ones use anywhere

java源程序（.java）在经过java编译器编译后成为java字节码文件（.class文件）。java程序要想执行，必须通过java虚拟机把字节码转换为机器码。

在不同的平台上，字节码是一样的但是虚拟机不同（例如：windows、Linux），运行在不同平台上的虚拟机将相同的字节码解释给不同平台上的cpu,这样，java程序就可以在不同的平台上运行了。

1.对象导论

OOP & OOA & OOL

OOP（Object-oriented Programming）面向对象程序设计。将问题空间中的元素及其在解空间中的表示称为对象。程序可以通过添加新类型的对象使自身适用于某个特定问题。

OOA（Object-Oriented Analysis）对问题域（按照面向对象的思想从业务角度对需求中描述的问题，描述问题的本质，确定问题中的对象。）和系统责任（开发系统应有的功能）进行分析理解。确定需求和系统能解决问题之间的关系（哪些能做），在此过程中，会面临需求的不断变化，需要开发人员与客户之间的交流。需求的稳定性（对象 > 属性 > 接口 > 功能）所以经常面向接口编程。

OOL（Object-Oriented Language）面向对象语言的三大特性封装、继承、多态。

类与对象

每个类的成员或元素都具有某种共性，比如每个银行账户都有结余金额。每一个对象都属于定义了特性和行为的某个特定的类，比如每个银行账户有不同的结余金额。

访问控制

访问控制的第一个存在原因是让程序员无法触及它们不应该触及的部分，这些部分对数据类型的内部操作来说是必须的，但并不是用户解决特定问题所需接口的一部分。对程序员来说也是一种服务，将重要的接口暴露给他们，不重要的隐藏起来。第二个原因是允许库设计者可以改变类内部的工作方式而不用担心会影响到使用它的程序员。

单根继承结构。在Java中，所有类都继承自Object，所有的类都有同一个根节点。单根继承结构保证所有对象都具备某些功能，因此在每个对象上都很容易执行某些基本操作。

基类型包含其所有导出类型所共享的特性和行为。基类型表示系统中某些对象的核心概念，导出类型可以表示此核心被实现的各种不同方式

有两种方法可以使基类与导出类产生差异：

1.添加新方法，基类方法不能满足所有需求，需要添加更多的方法。如果不添加新方法，子类和基类可以说是is A关系，添加了新方法后是is like A关系。

创建子类对象时是否创建了父类对象？

创建子类对象时,并没有创建父类对象,只是子类对象中有一个父类的亚对象，父类信息的存储依赖于子类，两者的地址是相同的。

子类继承父类哪些东西？

继承父类所有的属性和方法，虽然因为访问权限有些不能访问，但是都继承了，包括静态变量和静态方法也继承了。

多态

常常把一个对象不当做它所属的特定类型来对待，而是将其当做基类的对象来对待，这使得程序员可以编写出不依赖于特定类型的代码。这样的代码不会受添加新类型影响，而且添加新类型是扩展一个面向对象程序以便处理新情况的最常用方式，这种设计降低了软件维护的代价。

当视图将导出类型对象当做基类类型对象看待时，程序如何知道代表的是哪个基类类型？

绑定：将一个方法调用同一个方法主体关联起来称作绑定。

前期绑定：在编译过程中就已经知道这个方法到底是哪个类中的方法，即程序执行前进行绑定。

后期绑定：程序直到运行时才能够确定对象的地址，而不是在编译期间。

Java中除了static和final方法（private方法属于final方法）之外，其他所有方法都是动态绑定。static方法是因为不同对象调用的同一个方法不存在多态，final方法是该方法不允许被重写也不存在多态。

对象生命周期

在被称为堆的内存池中动态创建对象。Java提供了垃圾回收机制可以自动发现对象何时不再被使用，并继而销毁它，自动释放对象占用的内存。

异常处理

Java内置异常处理，并且强制使用。异常是一种对象，它从出错地点抛出，并被专门设计用来处理特定类型错误的相应的异常处理器catch，异常处理就像是与程序正常执行路径并行的、在错误发生时执行的另一条路径。

并发编程

为了提高并发量，我们使用多个线程完成不同用户的同一个请求操作。某个线程（任务）锁定某项资源，完成其任务，然后释放资源锁，使得其他任务可以使用这项资源。

Why Java？

如果Java仅仅只是众多的程序设计语言中的一种，为什么它促使计算机编程语言向前迈进了革命性的一步？

尽管Java对于解决单机程序问题非常有用，但更多的是它解决了WWW上的程序设计问题。

CS与BS架构：

C/S结构，即Client/Server(客户机/服务器)结构，是大家熟知的软件系统体系结构，通过将任务合理分配到Client端和Server端，降低了系统的通讯开销，可以充分利用两端硬件环境的优势。早期的软件系统多以此作为首选设计标准，市场份额逐渐减小。
B/S结构，即Browser/Server（浏览器/服务器）结构，是随着Internet技术的兴起，对C/S结构的一种变化或者改进的结构。在这种结构下，用户界面完全通过WWW浏览器实现，一部分事务逻辑在前端实现，但是主要事务逻辑在服务器端实现，形成所谓3-tier结构。

BS客户端编程

web最初的交互性完全由服务器提供，服务器产生静态页面，提供给只能解释和显示它们的客户端浏览器。基本的HTML只能将表单提交给服务器。这种提交动作通过web服务器的CGI传递，提交内容告诉CGI应该怎样如何处理它。最常见的动作是服务器端运行一个程序，几乎所有语言都可以编写该程序，Perl已经成为最常见的选择，然而Python已经对它产生了重大的冲击。然而构建在CGI上的程序可能会迅速变得过于复杂而难以维护，并且有响应时间过长的问题。比如web输入表单进行数据验证，按下提交按钮，数据被发送给服务器，服务器启动一个CGI程序来检查，发现错误，并将错误组装为一个HTML页面发送会浏览器，这个过程很慢。

问题的解决办法是客户端编程，客户端编程意味着web浏览器能够用来执行任何它可以完成的工作，使得返回给用户的结果更加迅捷，而且使得你的网站更加具有交互性。客户端编程最重要的一步是插件的开发，通过编写浏览器插件提供语言扩展，使得浏览器支持该编程语言。但并不是所有的客户端编程语言都是插件形式实现的，比如JavaScript。插件引发了浏览器脚本语言的开发，通过使用某种脚本语言，可以将客户端程序的源代码直接嵌入到html中，浏览器中解释这种语言的插件被自动激活。

Java applet通过applet进行客户端编程，applet是可以在浏览器中运行的小程序，它作为网页的一部分自动下载，但这种技术并没有达到当初它吹嘘的境界，最大的问题可能在于安装JRE所必须的10MB带宽（微软的IE浏览器中没有包含JRE这一事实封杀了applet的命运）。无论怎样Java applet没有得到大规模应用。后来微软相对于Java推出与之竞争的对手：.net平台（大致相当于JVM）和C#编程语言，但可能因为是后起之秀，没有良好生态，国内用的好像比较少。

既然Java有垃圾回收机制，为什么还需要finalize这个方法呢？

对finalize的需求限制到一种特殊情况：即通过某种创建对象方式以外的方式为对象分配了存储空间。那这种特殊情况是怎么回事呢？这种情况主要发生在使用本地方法的情况下，本地方法是一种在Java中调用非Java代码的方式。本地方法目前只支持C和C++，但它们可以调用其他语言写的代码，所以理论上可以调用任何代码。在非Java代码中，也许会调用C的malloc函数系列来分配存储空间，并且除非调用了free函数，否则存储空间将得不到释放，从而造成内存泄露，所以需要finalize这个函数来进行内存释放。

finalize被调用后，在下一次垃圾回收动作发生时，才会真正回收占用的内存。

7.内部类

可以将一个类的定义放在另一个类的定义内部，这就是内部类。迭代器设计模式是内部类的一个典型应用。

内部类起初看起来是一种代码隐藏机制：将类置于其他类的内部，但是，你将会了解到内部类远不止如此，它了解外围类，并能与之通信，而且你用内部类写出的代码更加优雅而清晰，尽管并不总是这样

最初，内部类可能看起来有些奇怪，而且要花些时间才能在设计中轻松地使用它们。对内部类的需求并非总是很明显的，但是在描述完内部类的基本语法和语义之后，10.8节就应该使得内部类的益处明确显现了。

内部类自动拥有对其外围类所有成员的访问权。这是如何做到的呢？当某个外围类的对象创建了一个内部类对象时，此内部类对象必定会秘密捕获一个指向那个外围类对象的引用。然后在你访问此外围类的成员时，就是用那个引用来选择外围类的成员。幸运的是，编译器会帮你处理所有的细节，但你现在可以看到：内部类的对象只能在与其外围类的对象相关联的情况下才能被创建（在内部类是非static类时）。构建内部类对象时，需要一个指向其外围类对象的引用，如果编译器访问不到这个引用就会报错。

.this 与 .new

如果你需要生成对外部类对象的引用，可以使用外部类的名字后面紧跟源点和this。这样产生的引用自动具有正确的类型，这一点在编译器就被知晓并收到检查，因此没有任何运行时开销。

有时可能想要告知某些其他对象，去创建某个内部类的对象。要实现此目的，必须在new表达式中提供对其他外部类对象的引用。

在拥有外部类对象之前是不可能创建内部类对象的。这是因为内部类对象会暗暗的连接到创建它的外部类对象上。但是，如果你创建的是嵌套类（静态内部类），那么它就不需要对外部类对象的引用

基本概念

C以及其他早期语言常常具有多种错误处理模式，通常会返回某个特殊值或者设置某个标志，并假定接受者将对这个返回值或标志进行检查，以判定是否发生了错误。如果的确在每次调用的时候都彻底的进行错误检查，代码会变得难以阅读，异常处理是把在正常执行过程中做什么事情与除了问题怎么办的代码相分离。

当抛出异常后，有几件事情会随之发生，首先，同Java中其他对象的创建一样，使用new在堆上创建异常对象。然后，当前的执行路径被终止，并且从当前环境中弹出对异常对象的引用。此时异常处理机制接管程序，并开始寻找一个恰当的地方来继续执行。这个恰当的地方就是异常处理程序（catch块）它的任务是将程序从错误状态中恢复，以使程序继续运行下去。

当把q传递给upcase方法时，实际传递的是引用的一个拷贝。其实，每当把String对象作为方法的参数时，都会复制一份引用，而被复制的引用其实一直待在单一的物理位置上，从未动过。

StringBuilder与 “+”

不可变性会带来一定效率问题。Java为String对象重载了+和+=，是Java中仅有的两个重载过的操作符，而Java不允许程序员重载任何操作符。

对于重载的“+”操作符，编译器自动引入了java.util.StringBuilder对象，用以构造最终的String，并为每个字符串调用一次StringBuilder的append方法，总计4次。最后调用toString方法生成结果。

上面两种构造字符串的方式通过反编译生成字节码，得出结论是对于“+”操作符StringBuilder是在循环之内构造的，没循环一次就会创建一个新的StringBuilder对象。因此当你为一个类编写toString方法时，如果字符串操作比较简单，那么直接用+操作符，如果需要使用循环，使用StringBuilder

14.注解

基本介绍

注解（annotation，也称为元数据），为我们在代码中添加信息提供了一种形式化的方法，使得我们可以在稍后某个时刻非常方便的使用这些数据。

注解使得我们能以将由编译器测试和验证的格式，存储有关程序的额外信息。注解可以用来生成描述符文件，甚至或是新的类定义，并且有助于减轻编写“样板代码的负担“。通过使用注解，我们可以将这些元数据保存在Java源代码中，并利用Annotation API为自己的注解构造处理工具，同时注解的有点还有：更加干净易读的代码以及编译期类型检查等。一般来说，主要还是需要程序员自己添加新的注解，并且按自己的方式使用它们。

java.lang中的注解

  • @Override，表示当前的方法定义将覆盖超类中的方法。如果你不小心拼写错误，或者方法签名对不上被覆盖的方法，编译器报错
  • @Deprecated，如果程序员使用了注解为它的元素，编译器报错
  • @SuppressWarings，关闭不当的编译器警告信息
  • Java还另外提供了四种注解，专门负责新注解的创建，将在稍后介绍它们。

每当你创建描述符性质的类或接口时，一旦其中包含了重复性的工作，那就可以考虑使用注解来简化与自动化该过程。注解（annotation）是在实际的源代码级别保存所有的信息，而不是某种注释性的文字，这使得代码更简介，更便于维护。通过使用扩展的annotation API，或外部的字节码工具类库，程序员拥有对源代码以及字节码强大的检查与操作能力。

NIO

JDK1.4的java.nio包中引入了新的Java IO类库（之前的叫做BIO），其目的在于提高速度，实际上，旧的IO包已经使用nio重新实现过，以便充分利用这种速度提高，因此即使我们不显式的用nio编写代码，也能从中受益。速度的提高在文件IO和网络IO中都有可能发生，我们在这里只研究NIO。

速度的提高来自于所使用的结构更接近于操作系统执行IO的方式：通道和缓冲器。我们可以把它想象成一个煤矿，通道是一个包含煤层（数据）的矿藏，而缓冲器是派送到矿藏的卡车，卡车载满煤炭而归，我们再从卡车上获得煤炭。也就是说，我们并没有直接和通道交互，我们只是和缓冲器交互，并把缓冲器派送到通道。通道要么从缓冲器获得数据，要么向缓冲器发送数据。

(个人觉得这个过程和网络编程中的包很类似)

旧IO类库中有三个类被修改了，用以产生FileChannel。这三个被修改的类是FileInputStream、FileOutputStream以及用于既读又写的RandomAccessFile。注意这些是字节操纵流，与低层的nio性质一致。Reader和Writer这种字符模式类不能用于产生通道。但是java.nio.channels.Channels类提供了实用方法，用以在通道中产生Reader和Writer
